## 2. Главная задача

Нужно реализовать во фронтенде **панель планировщика**, которая позволяет:

1. выбрать исполнителей через команды;
2. показать по выбранным исполнителям колонки;
3. показать тайм-сетку;
4. размещать в слотах объекты разных типов;
5. различать тип объекта визуально;
6. по клику на пустой слот запускать создание нужного объекта;
7. показывать ключевые данные объекта прямо в карточке без hover popover / popup;
8. быстро оценивать загрузку исполнителя на горизонте следующих дней через независимый скролл колонок;
9. передавать объект в другой слот того же исполнителя или другому исполнителю через модалку выбора слотов.

Типы объектов, с которыми должен работать новый планировщик:

* задача
* обращение
* событие

## 8. Что именно нужно реализовать

---

### 8.1. Пункт главного меню

Добавить в главное меню отдельный пункт:

**Планировщик**

Требования:

* отдельный route;
* встроить в текущую навигацию проекта;
* не ломать существующее меню;
* использовать текущий layout.

---

### 8.2. Экран планировщика

Создать экран/страницу модуля планировщика.

На странице должны быть:

1. compact toolbar / панель управления в одну строку;
2. выбор команд/исполнителей;
3. при необходимости — блок фильтров;
4. очередь;
5. доска с колонками исполнителей;
6. независимая подгрузка данных по колонкам;
7. обработка loading / error / empty state.

---

### 8.3. Выбор исполнителей

Дать пользователю возможность работать с исполнителями через команды.

Требования:

* использовать существующий инструмент выбора пользователей, который уже используется в задачах, внутри редактора команды;
* пользователь может создать команду, задать ей название и набрать в нее людей;
* по клику на команду planner переключается только на эту команду;
* через галочки можно добавлять несколько команд к просмотру одновременно;
* после выбора активных команд именно их исполнители становятся колонками планировщика;
* сохранять команды и активный выбор локально;
* не делать новый кастомный селектор, если в проекте уже есть рабочий.

---

### 8.4. Колонки исполнителей

После выбора исполнителей должны появляться колонки.

Требования:

* 1 колонка = 1 исполнитель;
* в шапке отображается имя исполнителя;
* допускается вывод доп. информации, если она уже есть в проекте: статус, загрузка, количество объектов;
* горизонтальный скролл обязателен;
* компактная верстка обязательна;
* стиль — по текущему проекту, а не по старому Planner.

---

### 8.5. Масштаб / размер слота

Нужно реализовать переключение размера/масштаба:

* 30 минут
* 1 час
* 2 часа

### Важно:

Согласованное изменение: отдельные режимы `день` и `неделя` в верхнем переключателе для текущей реализации не требуются.

Для текущей реализации принять следующие правила:

#### Режим “30 минут”

* день делится на получасовые слоты;
* объекты ставятся в конкретное время.

#### Режим “1 час”

* день делится на часовые слоты;
* объекты ставятся в конкретное время.

#### Режим “2 часа”

* день делится на двухчасовые слоты;
* объект ставится в ближайший двухчасовой интервал.

Дополнительно:

* каждая колонка имеет собственный вертикальный скролл;
* при прокрутке вниз подгружаются следующие дни только для этой колонки;
* planner не должен блокироваться целиком при подгрузке данных.

---

### 8.6. Клик по пустому слоту

При клике на пустой слот нужно показать промежуточную модалку выбора типа создаваемого объекта:

* задача
* обращение
* событие

После выбора типа:

* открыть **существующую** модалку создания соответствующего объекта;
* не писать новую форму создания с нуля;
* если возможно, передать начальные данные:

  * выбранный исполнитель;
  * дата;
  * время начала;
  * длительность/диапазон по умолчанию;
  * источник = планировщик.

Если у конкретной модалки часть initial values не поддерживается, это нужно:

* зафиксировать в документе знаний;
* не ломать существующую модалку ради “идеальной передачи всех полей”;
* внедрять только в допустимых рамках архитектуры текущего фронта.

### 8.7. Очередь

Для MVP реализовать очередь как **левую боковую панель**.

Это решение зафиксировать как базовое.

Требования:

* очередь находится слева от колонок исполнителей;
* в очереди лежат **неназначенные объекты**;
* очередь единая, но должна поддерживать фильтрацию по типам:

  * задача
  * обращение
  * событие
* желательно поддержать поиск;
* поддержать drag-and-drop из очереди в слот;
* вернуть объект из слота в очередь — предусмотреть архитектурно, даже если действие не войдет в первую итерацию для всех типов.

Почему именно так:

* это самый понятный диспетчерский UX;
* это ближе к референсу старого Planner;
* это не прячет очередь в лишние клики.

---

### 8.8. Карточки объектов в слоте

Планировщик должен понимать, какой тип объекта находится в слоте, и пользователь должен это сразу видеть.

Нужно сделать **единый каркас карточки**, но с конфигами по типу объекта.

#### Общая архитектура карточек:

* базовый контейнер карточки;
* общий слой layout;
* конфиг типа объекта;
* типоспецифичные поля;
* типоспецифичный попап.

#### Минимально различать через:

* цвет;
* иконку;
* типовой бейдж/метку;
* заголовок.

#### В карточке предусмотреть:

* тип объекта;
* название;
* время / диапазон;
* статус;
* краткие вторичные данные.

#### Не делать:

* хардкод только под один тип;
* разные карточки без общего каркаса;
* ручную кашу из if/else по всему компоненту.

Нужно сделать конфигурируемую схему, чтобы потом можно было добавлять новые типы объектов.

---

### 8.9. Карточка без hover popup / popover

Hover popover / popup для карточек не использовать.

Нужно заложить архитектуру:

* ключевые данные читаются прямо в карточке;
* действие переноса доступно с самой карточки через компактную inline-кнопку;
* по клику карточка открывает существующую сущность;
* отдельные поля по типам продолжают определяться из реального фронтенда и API.

#### Минимальные требования:

* задача: показать релевантные поля задачи прямо в карточке;
* обращение: показать релевантные поля обращения прямо в карточке;
* событие: показать релевантные поля события прямо в карточке.

Нельзя на этом этапе выдумывать поля вслепую.
Сначала найти реальные поля в текущем фронтенде и API.

---

### 8.10. Горизонт загрузки исполнителя

Это одна из главных функций.

Отдельный специальный режим `2 недели` с кнопкой в шапке колонки для текущей реализации не требуется.

Для текущей реализации принять следующую механику:

* у каждой колонки исполнителя свой вертикальный скролл;
* при прокрутке вниз подгружаются следующие дни только для этой колонки;
* planner не должен блокироваться целиком из-за загрузки одной колонки;
* loader показывается только на той колонке, которая догружает данные;
* пользователь должен быстро понимать загрузку исполнителя без отдельного режима переключения.

### Источник данных:

* нужно исследовать, есть ли API, которое уже умеет отдавать занятость по диапазону дат;
* если нет — собрать из имеющихся endpoint’ов;
* допускается независимая подгрузка по колонкам, если это не фризит весь planner;
* если backend не умеет удобный диапазон, зафиксировать это в документе и сделать управляемый фронтовый слой загрузки.

---

### 8.11. Передача объекта другому исполнителю

На карточке объекта должна быть кнопка/действие:

**Передать**

По нажатию открыть модалку переноса.

#### Модалка должна:

* показать исполнителей, которые выбраны в текущем планировщике;
* показывать это в виде матрицы: по горизонтали исполнители, по вертикали дни и слоты;
* показывать время прямо внутри ячейки слота;
* занятые слоты подсвечивать оранжевым, но разрешать выбирать;
* поддерживать выбор нескольких последовательных слотов через галочки;
* если пользователь выбирает непоследовательный диапазон, старый выбор должен сбрасываться до валидного непрерывного диапазона;
* поддерживать infinite-scroll по дням вниз в модалке вместо жесткого лимита в 14 дней.

#### Сценарий действия:

1. пользователь открывает перенос;
2. видит исполнителей и их слоты;
3. выбирает нужный слот или последовательный диапазон слотов;
4. объект переносится на того же или другого исполнителя в новый интервал.

### Для MVP принять:

* перенос делается **через модалку выбора слота**, а не только drag-and-drop;
* этим же сценарием должен покрываться перенос внутри того же исполнителя;
* главное — чтобы пользователь уже мог быстро передать объект туда, где есть место.

## 9. Ключевая аналитическая задача — диапазон слота для каждого типа объекта

Это критический пункт.

Нужно выявить и формализовать, что является диапазоном слота для:

* задачи
* обращения
* события

### Для каждого типа объекта определить:

1. поле начала;
2. поле конца;
3. если конца нет — как вычисляется длительность;
4. если есть только дата — как объект отображается на сетке;
5. как ведет себя объект без времени;
6. как задается слот по умолчанию при создании;
7. как объект переносится в другой слот;
8. допускается ли перенос между исполнителями;
9. какие серверные ограничения есть;
10. как обрабатывать all-day / многочасовые / многодневные объекты.

### Нельзя:

* захардкодить одинаковую логику для всех типов;
* просто считать “всё по часу”;
* просто привязать все типы к `start/end` без проверки реальных моделей.

Нужно сделать **слой адаптеров по типу сущности**.

Пример архитектурного подхода:

* `plannerEntityAdapters/taskAdapter`
* `plannerEntityAdapters/requestAdapter`
* `plannerEntityAdapters/eventAdapter`

Каждый адаптер должен уметь:

* получить display title;
* получить display status;
* получить executor;
* получить time range;
* получить all-day flag;
* получить данные для карточки;
* получить данные для inline-представления карточки;
* определить, можно ли переносить;
* построить payload переноса/создания/обновления.

## 13. Приемочные критерии

Функция считается выполненной, если:

### Навигация

* в главном меню есть пункт “Планировщик”;
* он открывает отдельную страницу модуля.

### Исполнители

* можно создавать команды и наполнять их через существующий селектор из задач;
* после выбора активных команд появляются колонки их исполнителей.

### Сетка

* есть рабочая сетка слотов;
* есть переключение масштаба:

  * 30 минут
  * 1 час
  * 2 часа

### Создание

* по клику на пустой слот открывается выбор:

  * задача
  * обращение
  * событие
* после выбора открывается существующая модалка соответствующего объекта.

### Отображение

* объект в слоте визуально показывает свой тип;
* пользователь может отличить задачу / обращение / событие;
* ключевые данные видны прямо в карточке без popover.

### Очередь

* есть левая боковая очередь;
* в ней отображаются неназначенные объекты;
* есть фильтр по типу;
* можно тянуть объект из очереди в слот.

### Длинный горизонт

* у каждой колонки свой вертикальный скролл;
* вниз по скроллу подгружаются следующие дни;
* данные догружаются только для активной колонки;
* planner не блокируется целиком.

### Передача другому

* на карточке есть действие “Передать”;
* модалка показывает исполнителей и их слоты;
* пользователь может выбрать один или несколько последовательных слотов;
* после выбора объект переносится.

### Документирование

* существует `docs/planner_research.md`;
* в нем зафиксированы:

  * найденные фронтовые точки интеграции;
  * найденные backend endpoint’ы;
  * диапазоны времени по типам объектов;
  * решения по спорным вопросам.

## 15. Дополнительные технические ориентиры из старого Planner

Использовать как ориентир по смыслу следующие идеи из старого Planner:

### По времени

* там уже была отдельная логика построения слотов;
* была логика определения предпочтительного диапазона;
* была логика раскладки пересекающихся карточек;
* была нормализация данных из backend в единый вид.

### По orchestration

* store получал данные;
* getters собирали UI-ready структуры;
* actions делали assign / move / unassign;
* UI только запускал действия и отображал состояние.

### По UX

* был toolbar;
* была очередь;
* были колонки;
* была карточка в очереди и карточка в слоте;
* были summary/filters/search.

Новый модуль должен взять оттуда:

* **принцип композиции**
* **принцип разделения данных и UI**
* **принцип расчета временной сетки**
* **принцип DnD-поведения**

Но не должен тащить:

* устаревший дизайн;
* логику только под заказы;
* старые визуальные компромиссы, если они не подходят текущему проекту.
