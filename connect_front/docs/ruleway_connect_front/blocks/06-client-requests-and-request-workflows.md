# Клиентские Обращения, Заявки И Согласования

## Разделы регламента

- `15. Клиентские обращения`
- `16. Заявки`

## Смысл блока

Обращения и заявки должны жить в одном цифровом контуре: фиксироваться, иметь историю обработки, быть привязаны к объектам и проходить через прозрачные статусы, исполнителей и согласования.

## Связанные фронтовые модули

- `HelpDesk`
- `Orders`
- `RequestApprovals`
- `Directories`
- `Notifications`
- `Documents`

## Что уже видно во фронте

- Есть `HelpDesk` с несколькими типами представлений.
- Есть `Orders` как оформитель заявок.
- Есть `RequestApprovals`.
- В `Directories` уже живут клиенты, позиции, категории, направления.

## Чего не хватает для соответствия регламенту

- Нет единой фронтовой карты “обращение vs заявка vs согласование”.
- Не видно сквозной истории перехода из обращения в заявку, встречу, задачу и согласование.
- Не хватает унифицированных статусных таймлайнов и журналов обработки.
- Не видно достаточного object binding для каждого шага пользовательского процесса.
- Отсутствует единый операторский UX для клиентского контекста.

## Функциональное направление развития

- Собрать общий request workflow layer поверх `HelpDesk`, `Orders`, `RequestApprovals`.
- Сделать единый timeline обработки и связанных артефактов.
- Дать операторам и руководителям единый экран клиентского кейса.
- Добавить прозрачные маршруты перехода обращения в задачу, заявку, встречу, документ и согласование.
