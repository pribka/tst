## Что нужно реализовать

### 1. Пункт главного меню
Добавь отдельный пункт главного меню:
**Планировщик**

Требования:
- отдельный route;
- текущий layout проекта;
- интеграция в существующую навигацию;
- учет текущих правил проекта по маршрутам и доступам.

### 2. Страница планировщика
Создай отдельную страницу модуля.

На странице должны быть:
- compact toolbar в одну строку;
- выбор команд/исполнителей;
- очередь;
- доска с колонками;
- состояния loading / empty / error;
- независимая подгрузка данных по колонкам без блокировки всего planner.

### 3. Выбор исполнителей
Используй существующий селектор пользователей из задач внутри редактора команды.

Требования:
- в toolbar пользователь работает не с голым списком исполнителей, а с командами;
- пользователь может создать команду, дать ей название и набрать в нее людей через существующий селектор пользователей;
- по клику на команду planner переключается на нее как на единственный активный набор;
- через галочки можно добавлять несколько команд к просмотру одновременно;
- команды и активное состояние хранить локально;
- не писать новый селектор пользователей, если текущий подходит.

### 4. Колонки исполнителей
После выбора исполнителей показывай колонки.

Требования:
- 1 исполнитель = 1 колонка;
- имя исполнителя в шапке;
- горизонтальный скролл;
- компактная верстка;
- стиль по текущему проекту.

### 5. Масштабы / размер слотов
Нужно поддержать режимы:
- 30 минут
- 1 час
- 2 часа

Согласованное изменение: отдельные режимы `день` и `неделя` в верхнем переключателе для текущей реализации не требуются.

Реализуй так:

- 30 минут: получасовые интервалы внутри дня;
- 1 час: часовые интервалы внутри дня;
- 2 часа: двухчасовые интервалы внутри дня;
- при прокрутке вниз в колонке подгружаются следующие дни только для этой колонки;
- planner не должен блокироваться целиком при подгрузке.

Все решения по этому вопросу запиши в `docs/planner_research.md`.

### 6. Клик по пустому слоту
При клике на пустой слот открой промежуточную модалку выбора типа объекта:
- задача
- обращение
- событие

После выбора типа:
- открой существующую модалку создания соответствующего объекта;
- не пиши новую форму создания;
- передай туда, если возможно:
  - исполнителя;
  - дату;
  - время начала;
  - длительность/интервал;
  - источник вызова = планировщик.

Если существующие модалки что-то из этого не поддерживают, зафиксируй это и адаптируйся к текущей архитектуре, не ломая ее.

### 7. Очередь
Для MVP делай очередь как левую боковую панель.

Требования:
- слева от колонок;
- содержит неназначенные объекты;
- единая очередь;
- фильтрация по типам:
  - задача
  - обращение
  - событие
- желательно поиск;
- drag-and-drop из очереди в слот.

Если возврат из слота обратно в очередь не покрывается сразу для всех типов, минимум заложи архитектуру под это.

### 8. Карточки объектов
Сделай единый каркас карточки объекта с конфигами по типам.

Нужно визуально различать типы:
- задача
- обращение
- событие

Используй:
- цвет;
- иконку;
- бейдж/метку;
- заголовок;
- диапазон времени;
- статус.

Не делай кашу из if/else по всему коду. Нужна конфигурационная схема.

### 9. Карточка без hover popup / popover
Hover popup / popover для карточек не использовать.

Требования:
- ключевые данные должны читаться прямо в карточке;
- действие переноса должно быть доступно с самой карточки через компактную inline-кнопку;
- по клику карточка открывает существующую сущность;
- реальные поля брать из кода и API, а не придумывать.

### 10. Горизонт загрузки исполнителя
Отдельный специальный режим `2 недели` с кнопкой в шапке колонки для текущей итерации не требуется.

Вместо этого:
- каждая колонка исполнителя имеет собственный вертикальный скролл;
- при прокрутке вниз подгружаются следующие дни только для этой колонки;
- подгрузка должна быть асинхронной и не блокировать остальные колонки;
- loader показывается только на той колонке, которая догружает данные.

Пользователь должен быстро понимать загрузку исполнителя без отдельного режима переключения.

### 11. Передать другому
На карточке объекта должно быть действие:
**Передать**

По нажатию открыть модалку переноса.

Модалка должна:
- показать исполнителей, выбранных в текущем планировщике;
- показать слоты по всем выбранным исполнителям в виде матрицы;
- показывать время прямо внутри ячейки слота;
- занятые слоты подсвечивать оранжевым, но разрешать выбирать;
- поддерживать последовательный выбор нескольких слотов через галочки;
- если пользователь выбирает непоследовательный диапазон, старый выбор должен сбрасываться до валидного непрерывного диапазона;
- поддерживать infinite-scroll по дням вниз в модалке вместо жесткого лимита в 14 дней;
- давать выбрать конкретный слот или последовательный диапазон слотов;
- после выбора выполнить перенос.

Для MVP перенос реализуй через модалку выбора слота. Этим же сценарием должен покрываться:
- перенос внутри того же исполнителя на другой день/время;
- перенос между разными исполнителями.

### 12. Слой адаптеров типов сущностей
Сделай адаптеры по типам объектов. Каждый адаптер должен уметь:
- получить тип;
- получить title;
- получить статус;
- получить исполнителя;
- получить start/end;
- определить all-day;
- собрать данные карточки;
- собрать данные для inline-представления карточки;
- определить возможность переноса;
- построить payload для изменения даты/исполнителя.

Если архитектура проекта подсказывает другое имя или форму организации — адаптируй под проект, но смысл должен сохраниться.
